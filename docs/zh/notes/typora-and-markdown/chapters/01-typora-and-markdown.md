# 第一章：像写作一样使用 Typora 与 Markdown

## 一、先理解一件事：Markdown 不是"炫技语法"，而是写作工具

很多人第一次接触 Markdown 时，都会冒出同一个疑问：既然 Word 已经能写文章，为什么还要再学一套看起来有点像代码的写法？这个问题不仅不幼稚，反而切中要害。如果一开始没想清楚 Markdown 到底解决了什么问题，后面的语法学习就很容易沦为机械记忆：标题是几个井号，粗体是几个星号，图片怎么插，表格怎么写。表面上会了不少，真正写文档时却仍然觉得别扭。

Markdown 出现的初衷，不是为了把写作变得更复杂，而是为了把写作重新拉回"内容优先"的状态。John Gruber 在最早的 Markdown 语法说明中强调，Markdown 的核心追求是"易读、易写"。换句话说，一个 Markdown 文本即使还没被渲染成漂亮的网页，单看纯文本本身，也应该足够清楚、足够自然，不会让人觉得像在读一堆难以辨认的标签。这个思想很重要——它意味着 Markdown 不是为了取代一切排版工具，而是为了让你在写作时尽量少被排版动作打断。

在传统富文本编辑器里，写作者经常会一边思考内容，一边忍不住处理样式：字号是不是太大了，段前距是不是没对齐，表格线是不是歪了，图片是不是应该再往左一点。这种工作方式不是不能用，但它会不断把注意力从"我要表达什么"拉向"它看起来是否足够像一份成品"。Markdown 的思路恰好相反。它鼓励你先把结构写清楚，再让软件根据结构去呈现样式。换句话说，Markdown 先关心"这是一段标题""这是一条列表""这是一段引用""这是一段代码"，然后才关心这些东西最终显示成什么样子。

也正因为如此，Markdown 特别适合知识笔记、技术文档、课程讲义、项目说明、博客草稿、论文草稿和长期积累型写作。当你的材料越来越多时，你会逐渐发现一个事实：真正难管理的并不是"怎么把某一页排得很好看"，而是"怎么让几十篇、几百篇文档都保持清晰、稳定、可迁移"。Markdown 在这件事上非常强，因为它的底层是普通文本文件。普通文本意味着可搜索、可版本管理、可迁移、可长期保存，也意味着你不会被某个单一软件的私有格式牢牢绑住。

## 一点五、为什么同一份 Markdown 在不同地方会显示得不一样

很多新手在刚开始学 Markdown 时，会遇到一个看起来很奇怪、其实非常正常的现象：同样一份 `.md` 文件，在 Typora 里看着很好，放到 GitHub 上时某些细节却不完全一样；换一个静态站点生成器后，目录、换行、表格、脚注、任务列表的表现又可能发生变化。于是就会得出一个误解，好像 Markdown 本身"不稳定"。

严格地说，不稳定的并不是 Markdown 这个想法本身，而是历史上 Markdown 的定义过于简短、过于强调直觉，导致很多边界情况没有被完全说死。CommonMark 规范之所以出现，正是因为原始 Markdown 说明虽然提出了清晰的设计哲学，却没有把许多解析细节完全写成机器可验证的规则。比如，子列表到底需要多少缩进，标题前是否必须空行，列表中的空行会不会改变结构，行末换行到底怎样才算硬换行，这些在不同实现里都曾出现过差异。

这件事对初学者非常重要，因为它直接决定了学习姿势。你不应该把 Markdown 当成一套"背下来以后永远百分之百一致"的绝对语法，而应该把它理解为一个有共同核心、但在不同生态中会有扩展和实现差异的文本写作体系。Typora 倾向于支持接近 GitHub Flavored Markdown 的常见扩展；GitHub 自己又有一套面向仓库协作场景的渲染规则；而 CommonMark 更像是在为这些行为提供一个更严格的共同地基。

因此，真正稳妥的学习方式并不是追求"记住所有平台的所有细节"，而是先掌握最稳的基础写法，再逐渐理解平台差异。比如，标题前后留空行、井号后面加空格、列表缩进写得清楚、图片尽量用相对路径、不要混用太多花哨写法，这些都属于在多个平台之间更不容易出问题的保守写法。你后面会发现，这种写法习惯不仅让文档更稳定，也让源文本本身更整洁。

## 二、Typora 为什么适合初学者

理解了 Markdown 的价值之后，第二个问题就来了：既然 Markdown 本质上是文本，那我是不是随便用个文本编辑器就行？理论上确实可以。你甚至可以用记事本、VS Code、Vim 或任何能保存纯文本的编辑器去写 Markdown。但是，理论上的"可以"，并不等于实践中的"适合起步"。

Typora 之所以长期被很多人视为 Markdown 入门的最好入口之一，原因就在于它把"源文本写作"和"所见即所得预览"结合得非常自然。你在 Typora 里输入 `# 一级标题`，它不会像很多双栏编辑器那样左边给你看原始代码、右边再单独给你看预览，而是直接在同一份文档里把它呈现成标题效果。当你把光标移回去时，又能看到它背后的 Markdown 结构。这种体验对新手尤其友好，因为它大大降低了"我正在写一堆奇怪符号"的陌生感。

更重要的是，Typora 并没有为了"好看"而彻底隐藏 Markdown。它不是把 Markdown 变成另一种 Word，而是用一种尽量平滑的方式让你逐步意识到：哦，原来这个标题背后是 `#`，原来这个复选框背后是 `- [ ]`，原来这个表格背后其实还是竖线和短横线。换句话说，Typora 不是替你绕过 Markdown，而是在帮你学会 Markdown。

根据 Typora 官方支持文档，它支持 GitHub Flavored Markdown 的许多常见扩展功能，例如任务列表、围栏代码块、表格、删除线、脚注、YAML Front Matter、目录、数学公式以及一部分增强语法。Typora 还提供了大纲视图、字数统计、主题切换、图片管理、导出 PDF/HTML、相对路径处理、数学公式与图像上传等实际写作中很有用的功能。这些能力组合在一起，使它不仅适合做"Markdown 学习器"，也适合做"长期写作工作台"。

## 三、安装 Typora 之前，你应该有的预期

如果你从来没用过 Typora，那么在安装之前最好先建立一个正确预期。Typora 不是一个复杂的排版系统，它的强项在于结构清晰、写作流畅和文档管理，而不是把视觉效果调到像 InDesign 那样精细。也就是说，你应该把它看成一个偏写作、偏知识生产的工具，而不是一个强调像素级布局的设计工具。

此外，Typora 从 1.0 版本开始已经不是无限期免费的 Beta 软件。官方文档明确说明，它提供试用期，正式长期使用需要购买授权。这一点提前知道会比较好，因为很多中文网络文章仍然停留在"Typora 是免费神器"的旧印象里。如果你准备把它作为长期主力工具，那么把它当作一款值得付费的软件来评估，会比抱着"先凑合用一阵"更现实。

安装完成后，你第一次打开 Typora，最值得做的不是急着敲一大堆语法，而是先观察它的工作界面：左侧是否显示文件树，右侧是否显示大纲，顶部菜单里有哪些和 Markdown、导出、图片、偏好设置有关的入口，偏好设置里有没有你需要开启的功能。这一步很像熟悉一间新的教室。你不必一开始就记住每个按钮，但你至少应该知道门、窗、黑板和开关大概在哪里。

## 四、Typora 的基本界面应该怎么看

第一次打开 Typora 时，很多人会被它的"安静"吓一跳。因为和一些工具栏密密麻麻的编辑器相比，Typora 显得非常克制。恰恰是这种克制，构成了它的写作优势。你真正需要熟悉的主要是四块区域：文档编辑区、文件侧栏、大纲视图以及菜单与偏好设置。

文档编辑区是你停留时间最长的地方。绝大多数工作都发生在这里。与双栏预览型编辑器不同，Typora 的文档区直接把 Markdown 渲染为接近最终阅读形态的效果。你在这里输入标题、段落、列表、代码块、表格和图片时，Typora 会在你输入的同时逐步帮你完成结构识别。对于初学者来说，这会产生一种非常重要的心理体验：你不会觉得自己在"写代码"，而更像是在"写内容"。

文件侧栏的价值，通常是在你写到第二十篇文档以后才真正显现出来。很多新手最开始只关心单篇文档怎么写，却忽略了文档集合怎么管理。实际上，一旦你开始把 Markdown 当成长期写作方式，你就会面对文件夹、命名、分类、章节组织、图片目录、草稿与定稿分离这些问题。Typora 的文件树能帮助你直接把"文档"理解为"目录中的文件"，这对长期管理非常关键。因为 Markdown 最终不是数据库里的黑盒对象，而是实实在在躺在文件夹里的 `.md` 文件。

大纲视图则对应"结构意识"。当一篇文章越写越长时，真正帮助你保持清醒的，往往不是段落本身，而是标题层级。如果你把标题写得有逻辑，大纲就会像骨架一样把文章撑起来。Typora 会自动根据标题层级生成大纲，你可以通过它快速跳转，也能反过来检查文章是否存在结构失衡。例如，某一节下面突然塞了十几个并列小节，而另一节只有两三句话，这通常意味着内容组织还不够稳。

至于菜单和偏好设置，不必一上来死记。你只要先知道几类重要功能放在哪里就行。比如，插入图片、表格、代码块、数学公式这类动作通常都能从菜单里找到；导出 PDF、HTML、Word 等格式也在菜单里；是否启用行内公式、上下标、高亮、图片处理策略、主题、自动编号等行为，则主要在偏好设置里控制。对于初学者来说，一个很实用的方法是：先靠菜单理解功能，再慢慢记快捷键。不要反过来一开始就逼自己背快捷键，那样很容易把学习体验变成负担。

## 五、写 Markdown 时，最先要掌握的是"结构"而不是"特效"

很多入门教程喜欢先列一串语法清单，标题、粗体、斜体、链接、图片、表格、代码、公式、脚注、任务列表、删除线、HTML……看起来好像很全面，但读完以后，新手常常并没有真正掌握写作。原因在于，Markdown 最核心的能力其实不是"会多少招"，而是"能不能把一篇内容写得结构分明"。

所以学习顺序一定要对。最先掌握的应该是标题、段落、列表、引用和代码块。因为这些元素决定了一篇文档最基本的骨架。粗体、斜体、删除线这类行内强调当然也重要，但它们更像修辞，不是骨架。你如果骨架没搭好，再多修辞也救不了一篇结构混乱的文档。

### 1. 标题：不是字号，而是层级

在 Markdown 里，标题的意义首先不是"看起来大"，而是"表示层级"。一级标题通常对应整篇文档的主题，二级标题对应主要章节，三级标题对应章节中的小节，再往下则是更细的层次。Typora 官方文档和 Markdown Guide 都明确采用井号 `#` 开头的 ATX 标题写法，这是今天最常见、兼容性也最好的方式。

例如，下面这几行：

```markdown
# Typora 入门
## 安装与界面
### 文件树与大纲
```

在视觉上会显示成不同大小的标题，但你更应该看到的是它们的结构关系：第二行是第一行的子级，第三行是第二行的子级。如果你把标题当成纯粹放大字体来用，就很容易出现层级混乱，比如一个三级标题前面没有二级标题，或者同一层标题承担完全不同粒度的内容。这些问题在短文里不明显，在长文里会非常致命。

Markdown Guide 还特别提到一个兼容性习惯：在井号和标题文字之间最好留一个空格，并且在标题上下留空行。这是非常值得养成的写法。很多渲染器虽然能容忍不规范写法，但跨平台、跨工具时，最稳的始终是规范写法。初学时就写得稳，后面迁移工具时会省下很多麻烦。

### 2. 段落：学会"空一行"比学会回车更重要

在 Word 里，很多人对段落的理解是"按一下回车就换行"。但在 Markdown 里，段落与换行不是同一个概念。Daring Fireball 的原始语法说明和 Typora 的官方说明都强调，段落是由一行或多行连续文本构成的，而不同段落之间通常靠空行分隔。

这意味着什么？意味着你写完一段话，如果只是普通换行但没有留空行，很多 Markdown 解析器会把它仍然当作同一段的一部分，而不是新段落。Typora 因为是面向写作体验的编辑器，会帮你更自然地处理很多输入动作，所以新手在 Typora 里常常意识不到"段落规则"的重要性。但如果你以后把文档放到 GitHub、静态网站、博客系统、笔记工具或其他解析器里，规范段落写法就会变得非常重要。

一个很好用的原则是：如果你希望读者感受到"这里开始说另一层意思了"，那就用空行分段。不要把所有内容堆成一整块，也不要每写一句话就起一个新段。段落的节奏，本质上是在安排思路的呼吸。

### 3. 换行：为什么你按了回车，却没有得到你以为的效果

初学 Markdown 时，最常见的困惑之一就是：我明明换行了，为什么显示结果还是连在一起？这个问题的答案其实正体现了 Markdown 的设计哲学。Markdown 默认认为"普通文本里的单次换行"常常只是作者在编辑时做的视觉折行，而不是想产生 HTML 里的真正换行。因此，真正的行内换行通常需要额外信号。

Markdown Guide 建议兼容性最好的方法有两种：一种是在行末保留两个空格再回车；另一种是在末尾使用 `<br>`。Typora 自己则提供了 `Shift + Enter` 这样的编辑器层面操作，让你更方便地产生单行换行效果。这里你要建立一个清晰观念：段落分隔和行内换行不是一回事。前者影响文本结构，后者影响段内呈现。理解这一点之后，你写诗歌、地址、签名、简历信息时就不会混乱。

### 4. 列表：不是为了"好看"，而是为了表达并列与顺序

列表在 Markdown 里有两大类：无序列表和有序列表。无序列表适合表达并列项，有序列表适合表达步骤、流程、次序。Typora 和 Markdown Guide 都支持常见的无序列表符号，例如 `-`、`*`、`+`，但实际写作里最好统一一种。我的建议很明确：日常正文统一用减号 `-`，这样最稳定、也最清爽。

例如：

```markdown
- 安装 Typora
- 创建第一篇 Markdown 文档
- 学会标题和段落
```

如果你要写操作步骤，则更适合用有序列表：

```markdown
1. 打开 Typora。
2. 新建文档。
3. 输入一级标题。
4. 保存为 .md 文件。
```

要注意的是，有序列表的数字在很多解析器里并不一定必须逐项严格递增，渲染器通常会自动计算编号。但从可读性和维护性来看，源文件里最好还是按真实顺序写。因为你最终维护的是一份"人也要读"的源文本，而不是只给机器看的输入。

列表真正难的地方，不是一级列表，而是嵌套结构。例如，一个步骤下面还要补充说明、再嵌套子步骤、再插入代码块，这时缩进就变得非常重要。Markdown Guide 明确提醒：在列表中继续插入段落、引用或代码块时，通常需要四个空格或一个 Tab 的缩进。很多人一开始以为 Markdown 很简单，结果一碰到复杂列表就全乱了，本质上就是没有把"缩进代表层级"这件事真正吃透。

### 5. 引用：它不是装饰框，而是"借用另一层声音"

引用块使用 `>` 开头。很多人第一次学会以后，觉得它只是一个灰色边框效果。其实引用块真正的意义在于，它为正文引入了一种"非主叙述"的声音。你可以用它放提示、定义、原文摘录、警告、注释、教师点评，甚至是对上文的补充说明。

例如：

```markdown
> Markdown 的设计重点不是排版特效，而是可读性与可写性。
```

一旦你理解引用的语义，你就不会滥用它。真正好的文档里，引用块通常只在"这里需要一种次级语气"时出现，而不是为了让页面看起来丰富一点就到处套一个框。

### 6. 代码块：什么时候该用，为什么不要滥用

如果你写的是技术文档、命令教程、程序笔记、配置说明，那么代码块几乎一定会频繁出现。Typora 官方文档说明，Typora 支持 GitHub Flavored Markdown 风格的围栏代码块，也就是三个反引号开始、三个反引号结束的写法。这比最早 Markdown 的"四个空格缩进代码块"更直观，也更适合今天的大多数场景。

例如：

````markdown
```bash
mkdir notes
cd notes
```
````

在 Typora 里，你只要输入三个反引号再回车，就会自动生成代码块输入区。若再加上语言标识，比如 `bash`、`python`、`javascript`，Typora 就能做语法高亮。

代码块的真正价值，在于把"需要按原样保留的文本"从普通叙述中剥离出来。命令、配置、程序片段、JSON、YAML、日志样例都属于这种情况。相反，如果你只是想在一句话里提到一个命令名或变量名，就更适合使用行内代码，而不是动不动起一个大块。好的文档写作，不是代码块越多越专业，而是该块则块、该行内则行内。

## 六、行内语法的作用，是精确强调，而不是到处涂颜色

当你已经能写出有基本结构的文档之后，才应该进入行内语法。行内语法的主要作用是让句子内部产生轻重、区分与提示。最常见的包括粗体、斜体、行内代码、链接和图片。

### 1. 粗体与斜体

粗体常用于强调关键概念，斜体常用于轻微强调、术语、书名、外来词或语气变化。Markdown Guide 与 Typora 官方说明都支持星号和下划线两种写法，但为了兼容性与一致性，通常建议优先用星号，尤其是行文中间夹着单词或变量时，更不容易出歧义。

例如：

```markdown
这是 **重点概念**。
这是 *轻度强调*。
```

真正需要警惕的是滥用强调。很多初学者一学会粗体，就想把一段里所有重要的词都加粗，结果读起来像满页都在喊重点。强调一旦过量，就会失去强调效果。你应当把粗体理解为"给读者目光打一个聚焦点"，而不是"每一句都刷存在感"。

### 2. 行内代码

行内代码使用单个反引号包裹，例如 `` `mkdir` ``、`` `README.md` ``、`` `Ctrl + Shift + P` ``。它特别适合出现在句子内部，用来表示命令名、文件名、参数、变量、路径或非常字面化的输入内容。

例如："请把文件保存为 `notes.md`，然后在终端中执行 `git add .`。"

行内代码有一种非常独特的气质：它在正文中告诉读者，"接下来这部分内容，不要把它当普通语义去理解，而要把它当作一个需要精确识别的字符串"。这在技术写作里极其重要。比如 `-` 和 `-`、`/` 和 `\`、大小写差异、扩展名差异，普通文本很容易看漏，而行内代码能大幅降低误读概率。

### 3. 链接

链接是 Markdown 最典型也最实用的能力之一。最常见的是行内链接：

```markdown
[Typora 官方支持文档](https://support.typora.io/)
```

在写作上，链接不只是"放一个网址"。它是文档建立外部连接的方式。一个成熟的知识库不会把自己写成封闭世界，而会在关键位置给出来源、扩展阅读、官方文档和参考资料。Typora 也支持引用式链接，不过初学阶段可以先把行内链接用熟。引用式链接更适合长文档中多次复用同一 URL，或者在正文里保持更清爽的视觉效果。

### 4. 图片

图片语法看起来和链接很像，只是在前面多一个感叹号：

```markdown
![一张示意图](./images/example.png)
```

但真正重要的并不是这行语法本身，而是你要理解 Markdown 中的图片并不是"嵌入进文件内部的数据"，而是"对一个图片文件的引用"。Typora 官方的 Images 文档对此解释得非常清楚：Markdown 文件本身是纯文本，所以图片通常通过 URL、绝对路径或相对路径被引用进来。

这件事的影响非常大。因为它意味着，只要你把图片路径管理得好，Markdown 文档就会非常可迁移；反过来，如果你今天一张图拖到桌面，明天一张图来自下载目录，后天一张图又在微信临时目录，那么等你把整套文档换台电脑时，十有八九会出现一片红叉。

因此，从一开始就建议你养成一个工作习惯：每篇教程或每个项目目录里，都建立相对稳定的图片文件夹，比如 `images/` 或 `assets/`，然后让 Typora 在插入图片时尽量复制到这个目录中，并使用相对路径。这样未来无论是 Git 管理、同步到云盘、发给别人，还是发布到 GitHub，都会轻松很多。

## 七、Typora 最值得新手掌握的几项实战能力

如果说 Markdown 语法解决的是"文档怎么写"，那么 Typora 的很多功能解决的是"文档怎么真正用起来"。下面几项能力非常值得在入门阶段就练熟，因为它们直接影响你以后的工作流质量。

### 1. 用 Typora 新建并保存第一篇文档

新手最容易忽略的一点，就是尽早把"草稿"变成"文件"。很多人会打开 Typora 随手写，写了半天还没保存，结果图片路径、文件位置、相对目录、导出位置全是漂浮状态。正确的做法是：一开始就先创建一个文件夹，再在这个文件夹里保存你的文档。

例如，你打算写一套关于写作工具的资料，那么可以先建立这样一个目录：

```text
writing-course/
├── docs/
├── images/
└── 01-typora-and-markdown.md
```

然后在 Typora 里打开或保存 `01-typora-and-markdown.md`。这样做的好处是，整篇文章从一开始就有了明确的归属目录。未来插图、导出、版本管理、同步与发布，都会围绕这个目录展开。很多后期的混乱，实际上都源于前期文件落点太随意。

### 2. 学会使用大纲，而不是只靠滚动条

只用滚动条定位长文，是一种非常低效的习惯。文章一旦超过几千字，你就应该依赖大纲。因为滚动条只告诉你"位置"，而大纲告诉你"结构"。

在 Typora 中，当你建立了清晰的标题层次后，大纲会自动生成。此时你应该经常做两件事。第一，利用大纲跳转到不同章节，而不是来回滚屏。第二，把大纲当成自检工具，观察某一节是不是过长、某一层是不是断裂、是否存在标题命名重复、粒度失衡等问题。长期这样写，你会逐渐形成"先想骨架，再填内容"的写作习惯。

### 3. 学会设置图片处理策略

Typora 官方关于图片的文档非常详细，其中最值得新手注意的是"插入图片时怎么处理源文件"。默认情况下，Typora 可以直接引用图片原路径。但这往往不够稳。更适合长期知识库的做法，是打开图片插入相关设置，让 Typora 在你拖拽、粘贴或插入本地图片时，把图片复制到指定目录。

一旦你这样设置，文档和素材之间的关系就会稳定很多。对个人长期写作来说，这几乎是从"随手记"走向"可维护资料库"的分水岭。你会发现，好的知识管理从来不是靠多高级的软件，而是靠一开始就形成稳定的文件组织习惯。

### 4. 学会导出，但不要过早依赖导出修修补补

Typora 的导出能力很强，可以导出为 PDF、HTML 甚至其他格式。很多新手会特别兴奋，觉得自己终于能"一键变成正式文档"。这个兴奋可以理解，但我更建议你把导出看成"最后一步"，而不是"边写边补救版式的工具"。

真正成熟的流程应该是：先把 Markdown 结构写清楚，再在导出时让样式系统接管版面。换句话说，导出是结果，不是拐杖。如果你每次写文档都依赖导出前的大量手工修修补补，那通常说明前面的结构设计还不够清晰。

### 5. 学会主题与自定义，但入门阶段不要沉迷换皮肤

Typora 有主题系统，这很讨喜，因为它会让写作界面更有风格感。官方还提供了主题和一部分 CSS 自定义相关能力，甚至可以实现标题自动编号等高级效果。比如 Typora 官方支持文档中就给出了用 CSS 实现标题自动编号的方案。

不过，初学阶段我并不建议你把大量时间花在"哪个主题更好看"上。因为主题和配色是使用体验的优化，不是写作能力的核心。最先应该稳定的是：目录结构、文件命名、标题层次、图片路径、代码块使用习惯和导出习惯。等这些东西都稳定以后，再去折腾 CSS，自定义目录编号、页面宽度、字体、代码高亮，才是锦上添花，而不是本末倒置。

## 八、如何真正开始写第一篇 Markdown 文章

现在，我们把所有观念收束成一个非常实际的问题：如果你今天第一次用 Typora 写 Markdown，第一篇文章到底应该怎么开始？

最好的方式不是对着语法表硬抄，而是先挑一个非常小、非常具体、而且你熟悉的主题，例如：

- 我的阅读笔记工作流
- 如何整理课程作业
- 怎样备份电脑资料
- 如何配置一个新项目目录

然后按下面的思路去写：

- 先写一个一级标题，作为文章主题。
- 接着写一小段导言，告诉读者这篇文章要解决什么问题。
- 然后安排三到五个二级标题，每个二级标题只负责一个明确小主题。
- 每个小主题下面先写完整段落，而不是急着往里面塞各种格式。
- 只有当你确实需要强调重点、展示命令、插入图片、罗列步骤时，才使用相应语法。

这种写法看起来朴素，却非常有效。因为它逼着你优先把文章讲明白，而不是优先把页面弄漂亮。很多人之所以一直学不会 Markdown，不是因为语法太难，而是因为一开始就把注意力放错了地方。

下面是一份很适合新手模仿的最小文章骨架：

```markdown
# 我的第一篇 Markdown 文章

这篇文章用来练习 Typora 与 Markdown 的基本写法。我会先说明写作目标，再分几个部分介绍实际做法。

## 为什么我要写这篇文章

这里写一段完整文字，解释动机。

## 我准备了什么

这里可以用列表，也可以直接写成段落。

## 实际步骤

这里适合写有序列表，如果涉及命令，就插入代码块。

## 写完之后我得到了什么

这里做一个总结，说明结果与收获。
```

你会发现，这份骨架一点也不复杂，但它已经包含了 Markdown 写作中最关键的思维：主题、段落、章节、步骤、总结。只要你愿意用这样的骨架重复写上几篇，语法会在过程中自然熟悉，而不是靠死记硬背灌进脑子里。

## 九、一个完整的入门示范：从空白页到一篇像样的文档

下面我们做一次更接近真实写作的示范。假设你要写一篇《如何整理课程笔记》的短文。你可以在 Typora 里这样展开。

先输入标题：

```markdown
# 如何整理课程笔记
```

接着写导言，不要太长，但要说明问题：

```markdown
很多人记了很多课程笔记，但过一段时间后就很难再找到需要的内容。本文想介绍一种更容易长期维护的课程笔记整理方式。
```

然后设置三个二级标题：

```markdown
## 先按课程建立文件夹
## 再按主题拆分文档
## 最后统一管理图片与附件
```

到了这里，这篇文章的骨架其实已经出来了。接下来你再在每一节下面补内容。如果某一节里需要列出步骤，就用有序列表；如果需要展示目录结构，就用代码块；如果需要说明一个关键概念，就用粗体强调；如果需要展示一张示意图，就插入图片。

例如，在"先按课程建立文件夹"这一节下，你可以给出一个目录结构：

````markdown
```text
courses/
├── math/
│   ├── 01-limit.md
│   └── images/
├── physics/
└── history/
```
````

在"最后统一管理图片与附件"这一节下，你则可以说明为什么推荐相对路径，以及 Typora 怎样帮助你复制图片到指定目录。这样一篇文档写下来，读者会觉得你是在认真教他建立工作方式，而不是只在展示几个语法动作。

## 十、图片、路径与文件夹：这是新手最容易掉坑、也最该尽早学会的一课

很多人学习 Markdown 时，注意力几乎全部放在标题、粗体、列表和代码块这些"看得见"的语法上，却把图片和路径当成小问题。结果写第一两篇文章时还没什么感觉，等文档数量一多，或者准备换电脑、同步到 GitHub、导出到别的地方时，问题就会集中爆发：图片找不到、路径全断、文档能打开但配图一片红叉、别人拿到文件夹后看不到任何插图。

这类问题之所以频繁出现，是因为很多人实际上还没有真正理解 Markdown 文件和资源文件之间的关系。Markdown 文档不是一个把一切都装进内部的封闭盒子。它本质上是一份文本，而图片、附件、插图、封面、示意图通常都在外部文件里，通过路径和链接被引用进来。正因如此，Markdown 文档写得稳不稳，往往不只取决于语法会不会写，还取决于你的文件系统是否有秩序。

Typora 官方关于图片的文档说得很清楚：图片可以来自网络地址，也可以来自本地绝对路径或相对路径；插入图片时，Typora 既可以直接引用原始路径，也可以复制图片到指定目录，再把文档中的链接改成相对稳定的引用方式。对于临时记录来说，直接拖一张图进来可能已经够用；但对于教材、笔记库、课程资料和任何准备长期维护的仓库来说，更稳的做法几乎总是把图片归档到当前文档所属目录或当前分册所属目录里。

例如，如果你正在写一册教材，那么比起让图片散落在桌面、下载目录和聊天缓存中，更合理的结构往往是这样的：

```text
books/
└── typora-and-markdown/
    ├── chapters/
    │   └── 01-typora-and-markdown.md
    └── images/
        ├── chapter-01-figure-01.png
        └── chapter-01-figure-02.png
```

当你采用这种结构时，文档中的图片引用就可以写成相对路径。相对路径的价值不只在于"看起来简洁"，更在于它会跟着整个目录一起移动。只要整册教材的目录结构不乱，你把这个文件夹复制到另一台电脑、同步到云盘、推到 GitHub，甚至发给别人，对方通常都能看到同样的图片。反过来，如果你在 Markdown 中引用的是某个系统目录下的绝对路径，例如桌面或下载目录，那么这份文档基本上只能在你自己的当前机器上正常工作。

GitHub 官方文档也特别强调，在仓库内部引用文件和图片时，优先使用相对链接而不是绝对链接。原因非常直接：相对链接跟仓库结构天然绑定，切换分支、复制仓库、别人 clone 下来之后，链接仍然更容易保持可用。如果你把仓库内部文档写成一堆绝对 URL，那看起来像是能用，但它失去了 Markdown 最宝贵的一部分能力--可迁移。

所以，对刚入门的人来说，我很建议尽快建立一个简单但稳定的原则：文档写在哪里，配套图片最好也归档到和它有明确关系的位置；引用能用相对路径，就尽量不要用绝对路径；如果 Typora 提供"插入图片时复制到指定目录"的设置，就尽量把它设置好，而不是每次随缘拖图。很多人把这类问题看成"小整理"，实际上这恰恰是把 Markdown 从"会写几个符号"提升到"会管理一套文档"的关键一步。

## 十一、为什么一定要考虑 GitHub：因为教材不是只在你电脑里活着

如果你把 Markdown 只当成个人临时记事工具，那么你确实可以长期不管 GitHub，不管平台差异，不管公开展示。但只要你准备把内容做成教材、文档、知识库、课程资料，GitHub 视角迟早会变得重要。因为它代表的不只是一个网站，而是一种很典型的"公开仓库阅读场景"：别人不是在你的 Typora 里看文章，而是在浏览器里、在仓库页面里、在一个有自己渲染规则的平台上阅读你的内容。

GitHub 官方关于 Markdown 的文档给了一个很有价值的提醒：很多写法在本地编辑器里看着没问题，但放到仓库页面时，真正决定展示效果的是 GitHub 自己的渲染逻辑。比如标题会自动生成锚点，多个标题会在界面里形成 outline，仓库内文件和图片更推荐使用相对链接，`.md` 文件中的普通单次换行并不会总被解释成你想象中的断行。换句话说，教材一旦进入发布场景，它就不再只是"你看起来舒服就行"，而要面对"读者怎样看到、怎样跳转、怎样引用、怎样继续阅读"的问题。

这也是为什么我不建议把 Typora 和 GitHub 看成两件互不相干的事。Typora 更像是你的写作台，GitHub 更像是你的展示台。写作台的作用，是帮助你更流畅地把内容组织起来；展示台的作用，是把这些内容稳定地交到读者手里。一个真正成熟的 Markdown 工作流，不应该只有前者而没有后者。

对教材写作者来说，这个意识会直接影响目录写法。比如，当你在根目录写总 README 时，它不应该只是给自己看的开工记录，而应该承担"让第一次来到仓库的人迅速理解这里是什么"的作用。分册 README 则应该让读者明白这一册讲什么、适合谁读、从哪里开始读。章节之间的跳转链接，也不应该只是为了自己方便，而应该服务于整个阅读路径。你一旦从"自己写"转向"别人也要读"，Markdown 的很多看似细小的规范就会一下子变得非常现实。

也正因为如此，我建议任何准备把 Markdown 内容做成教材的人，尽早培养一种发布意识：你写的不是一份只在本机编辑器里临时成立的稿子，而是一份未来可能在仓库、网页、导出 PDF、静态站点和不同设备上被反复阅读的文档。带着这种意识去写，很多决定会更稳，比如标题层级会更克制，目录结构会更清楚，链接会更规范，图片路径会更有秩序。

## 十二、不要把快捷键当表格去背，而要把它们变成写作动作的一部分

很多软件教程讲快捷键时，喜欢直接甩一大张表：新建是什么、保存是什么、标题是什么、代码块是什么、引用是什么、全屏是什么。读者看完以后通常会有一种错觉，好像自己已经"掌握得差不多了"。但真正开始写文档时，手还是会回到鼠标，菜单还是要一层层点，输入节奏还是会被不断打断。原因不复杂：快捷键不是靠背下来掌握的，而是靠在真实动作里反复使用，最后变成肌肉记忆。

Typora 官方快捷键文档很完整，里面列了文件、编辑、段落、格式和视图相关的常用操作。对初学者来说，最值得先熟悉的并不是全部，而是那些会直接改变写作节奏的少数几个动作。比如，标题等级切换、代码块、引用块、无序列表、有序列表、插入图片、插入链接、新段落和单行换行，这些都是在写教材正文时非常高频的动作。

这里有一个很实用的学习顺序：

- **第一步**：不要试图一次掌握所有快捷键，而是先盯住三到五个你每篇文档都会反复使用的动作。
- **第二步**：在一周左右的时间里强迫自己优先用这些快捷键，而不是用鼠标菜单。
- **第三步**：当这几个动作已经变得自然，再把下一组动作加进来。

这样学的好处是，快捷键会逐渐嵌进你的写作过程，而不是停留在脑子里的死知识。

例如，假设你今天主要在练标题、列表和代码块，那你就可以有意识地把"新建标题""插入有序列表""插入代码围栏"当作今天的重点动作。你会发现，一旦这些动作变得自然，你写作时的注意力就更不容易被界面操作打断。Markdown 原本就是一种强调思路连续性的写作方式，而快捷键其实是在帮助你保护这种连续性。

但是，这里也要避免另一个极端：不要把熟练度理解成"像打游戏一样追求手速"。教材写作的本质仍然是组织知识，而不是证明自己对工具有多熟。快捷键的意义，不是让你显得专业，而是让你减少无意义的切换成本，把更多精力留给结构、表达和判断。真正有效的工作流，不是你会多少快捷键，而是你在需要的时候，能不能自然地调出最合适的动作。

## 十三、Typora 与 Markdown 最常见的误区

入门阶段最常见的误区，不是不会写，而是用错了理解方式。这里最典型的几类问题，其实可以直接归纳为下面四种：

- **误区一：把 Markdown 当成"轻量版 Word"**
  一旦这样想，你就会不断追问为什么它不能自由拖动图片、为什么不能像排版软件那样任意改局部样式、为什么表格没有想象中那么随心所欲。实际上，Markdown 的价值从来不在像素级自由，而在结构化表达与长期可维护性。如果你追求的是杂志排版级控制，Markdown 根本不是为那个目标设计的。

- **误区二：把 Typora 当成"完全不用学语法的可视化编辑器"**
  这种想法短期看很舒服，长期却会让你卡住。因为你一旦离开 Typora，就会发现自己不懂源文本，不知道为什么这里能显示、那里不能显示，也不知道为什么同样一篇文档换个平台后样式不同。真正高效的方式，是借 Typora 的所见即所得体验进入 Markdown，但最终仍然理解 Markdown 的基础规则。

- **误区三：过早追求高级功能**
  比如一开始就执着于脚注、公式、自动目录、YAML、主题 CSS、自定义导出、图床上传，却没有把标题、段落、列表、代码块和图片路径管理写稳。结果往往是：看起来学了很多，真正写一篇完整文档时却仍然凌乱。基础不稳时，高级功能只会增加噪声。

- **误区四：忽略文件系统**
  Markdown 文档不是孤立存在的。它与文件名、目录、图片路径、相对引用、导出位置、同步方式紧密相关。如果你不把 Markdown 当成"文件系统中的文档"，而只把它当成"编辑器中的一页纸"，后面一定会在迁移、同步和发布时吃苦头。

## 十四、当你开始熟练之后，Markdown 会改变你的写作方式

一开始，Markdown 看起来像是在增加一层规则；但真正用熟之后，你会发现它其实是在帮你减负。因为它不断提醒你：先把结构想明白，再把内容写清楚，最后再考虑呈现。你不再需要一边写一边担心字号、字体、段距和对齐方式；你开始更多思考标题是否准确、段落是否完整、例子是否合适、结构是否流畅。

这也是为什么很多长期写作者、程序员、知识管理者、研究者会最终稳定在 Markdown 生态里。不是因为 Markdown 更酷，而是因为它让写作更接近思考本身。你在编辑器里写的是普通文本，在渲染器里看到的是清晰排版，在版本控制里保存的是可比较的历史，在 GitHub 或网站上发布的是结构稳定的内容。写作、保存、同步、协作、发布这一整条链，第一次被一种极其朴素的文本格式连了起来。

## 十五、练习：请你自己写一篇三节小文

如果你真的想把这一章学会，最好的方法不是再看十篇"Markdown 语法大全"，而是立刻在 Typora 里写一篇三节小文。题目可以很简单，例如《我的桌面整理方法》《如何准备一次考试》《我怎样备份手机照片》。要求可以明确成下面几条：

- 先给文章写一个一级标题。
- 然后写一段不少于三句话的导言。
- 接着写三个二级标题，每节至少写一个完整段落。
- 在其中一节中插入一组有序列表。
- 在另一节中插入一个代码块或者目录结构示例。
- 最后在结尾写一段总结，说清楚这篇文章想传达的中心意思。

当你完成这篇小文时，你会发现自己已经不知不觉把标题、段落、列表、代码块和结构组织都练过一遍了。真正的入门，不是知道每条语法，而是独立写出一篇像样的文档。

## 十六、下一步该学什么

当你已经能稳定写出结构清晰的 Markdown 文档后，下一步最值得继续学习的，不是把所有高级语法一次性吃完，而是进入更接近真实工作流的主题。例如，怎样管理图片与相对路径，怎样组织多篇文档的目录结构，怎样写表格和公式，怎样把 Markdown 与 Git、GitHub、静态站点、知识库系统结合起来。那时你会发现，Markdown 学到后面，早已不是"会不会写井号和星号"的问题，而是在搭建自己的知识生产系统。

## 附录二：建议你立刻养成的三个习惯

- **先定文件落点，再开始写**：任何正式文档都先保存到确定的文件夹里，再开始写。
